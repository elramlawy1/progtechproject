ğŸ¯ DESIGN PATTERNS - MEMORIZE THESE!

1. STRATEGY PATTERN (ai package):
   Interface: AIStrategy
   Implementation: RandomAIStrategy  
   Client: AIPlayer
   Why? Easily swap or add new AI strategies without changing code
   
2. SINGLETON PATTERN (DatabaseManager):
   Only ONE instance exists
   Method: DatabaseManager.getInstance()
   Why? One database connection for whole app, avoid conflicts
   
3. FACADE PATTERN (Game class):
   Complex operations hidden behind simple interface
   Example: makeMove() handles validation + update + win check
   Why? Simplifies usage, UI doesn't need to know board internals

Q: "Explain your architecture"
A: "I used a layered architecture:
    - Model layer: Board, Game, Move (business logic)
    - AI layer: Strategy pattern for extensible AI
    - Database layer: Singleton for connection management
    - UI layer: CommandLineInterface for user interaction
    Each layer has single responsibility, loose coupling."

Q: "How does win detection work?"
A: "After each move, Board.checkWinner() checks 4 directions from that
    position: horizontal, vertical, and both diagonals. For each direction,
    checkDirection() counts consecutive pieces. If count >= 5, we have a winner.
    It's in Board.java lines 175-205."

Q: "Why these design patterns?"
A: "Strategy: AI can be swapped/extended without changing AIPlayer
    Singleton: Only one DB connection needed, prevents conflicts
    Facade: Game.makeMove() hides complex board operations from UI"
